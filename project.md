# 情侣生活记录 iOS 应用实现方案

## 1. 项目目标

### 1.1 产品定位

这是一款面向情侣使用的 iOS 应用，核心目标是共同记录生活、安排计划、关注健康状态，并在长期使用中沉淀为两个人的生活档案。

产品不应只停留在“打卡工具”层面，而应兼顾以下三个方向：

- 记录：把日常事件、身体状态、习惯行为沉淀下来
- 协作：让双方可以共享计划、提醒和进度
- 洞察：通过日历和运动健康数据生成更清晰的生活视图

### 1.2 核心原则

- 优先做 iOS 原生体验，充分接入 Apple 生态
- 先保证记录和计划能力稳定，再逐步做美化与分析
- 数据模型从一开始就预留扩展空间，避免后续推倒重来
- 共享能力要支持“个人私密数据”和“情侣共享数据”并存

## 2. 产品范围

首期产品包含三个核心板块：

- 日历
- 待办清单
- 运动

未来可扩展板块：

- 纪念日与关系里程碑
- 情绪记录
- 饮食记录
- 旅行记录
- 照片墙 / 时光轴
- AI 总结与生活洞察
- 习惯养成与挑战机制

## 3. 功能设计

## 3.1 日历板块

### 目标

用日历作为统一时间入口，承载生活记录、健康事件和计划视图。

### 首期支持记录项

- 喝水
- 排便
- 月经
- 睡眠
- 日常活动
- 其他自定义记录

### 记录方式

- 点击某一天进入详情页进行记录
- 支持快捷打卡
- 支持同一天多条记录
- 支持文字备注、标签、时间段、状态值
- 支持未来补录和过去补录

### 日历视图建议

- 月视图：查看整体分布和关键标记
- 周视图：适合查看密集记录和计划安排
- 日视图：查看完整时间线

### 展示设计

- 用统一图标体系展示不同记录类型
- 月经、睡眠、排便等使用不同颜色或形状区分
- 支持“我”和“TA”的双人视角切换
- 支持合并视图，查看情侣双方同一天的活动与状态

### 未来扩展位

- 体重
- 体温
- 心情
- 用药
- 饮食
- 症状追踪
- 生理周期预测

## 3.2 待办清单 / 任务计划板块

### 目标

构建从日计划到年计划的多层级任务系统，并与 iOS 日历联动。

### 首期任务层级

- 日计划
- 周计划
- 月计划
- 年计划

### 每个任务建议字段

- 标题
- 描述
- 开始时间
- 截止时间
- 是否全天
- 优先级
- 所属层级
- 状态
- 负责人
- 是否共享
- 重复规则
- 标签
- 关联日历事件 ID
- 关联目标 / 项目 ID

### 功能设计

- 支持个人任务与情侣共享任务
- 支持任务拆分为子任务
- 支持任务完成、延期、取消
- 支持将任务同步到 iOS 日历
- 支持从 iOS 日历导入事件生成任务
- 支持计划视图和列表视图切换

### 与 iOS 日历联动方式

基于 EventKit 接入系统日历：

- 读取系统日历事件
- 创建应用内任务对应的系统日历事件
- 更新任务时同步更新系统事件
- 删除任务时可选择保留或删除系统事件

### 设计建议

不要让“任务”和“日历事件”在底层完全耦合。建议定义为两个独立实体，用映射关系连接。这样未来即使接入更多第三方日历或提醒事项，也不会影响核心任务模型。

### 未来扩展位

- 项目管理模式
- 双人目标管理
- 周报 / 月报自动生成
- 任务统计与完成率分析
- 模板计划

## 3.3 运动板块

### 目标

接入 Apple 健康和健身能力，对基础运动指标进行更美观、更适合情侣共享场景的展示。

### 首期接入能力

- 步数
- 步行 / 跑步距离
- 活动能量消耗
- 站立时长或站立次数
- 运动时长
- 心率
- 睡眠摘要

### 技术接入

- HealthKit：读取健康数据
- 视情况接入 Workout 相关数据
- 使用本地缓存减少频繁读取 HealthKit 的开销

### 展示建议

- 日 / 周 / 月数据切换
- 与伴侣对比视图
- 趋势折线图 / 柱状图
- 达标提示和轻量鼓励机制
- 睡眠、步数、运动时间组合卡片

### 注意事项

- HealthKit 权限必须按需申请，不要一次申请过多
- 明确标记哪些数据来自系统同步，哪些来自手动录入
- 如果涉及月经与健康数据整合，必须提供足够明确的隐私说明

### 未来扩展位

- 共同运动挑战
- 周期与运动状态联动分析
- Apple Watch 配套体验
- 智能提醒
- 健康趋势洞察

## 4. 用户体系与共享模型

### 4.1 基础关系模型

建议产品基于以下结构设计：

- User：单个用户
- CoupleSpace：情侣共享空间
- Membership：用户与共享空间关系

### 4.2 数据归属

每条数据都建议具备以下归属字段：

- ownerUserId：创建者
- visibility：private / shared
- coupleSpaceId：所属情侣空间，可为空

这能解决两个关键问题：

- 用户有些记录只想自己看
- 有些任务或活动需要双方共享

### 4.3 隐私边界

尤其是月经、排便、睡眠等敏感数据，建议支持：

- 私密记录
- 仅汇总共享
- 完全共享

不要默认所有健康相关信息都对伴侣可见。

## 5. 信息架构建议

底部 Tab 建议如下：

- 首页
- 日历
- 计划
- 运动
- 我的

### 首页内容建议

首页作为情侣共同生活的总览页，聚合展示：

- 今日任务
- 今日记录摘要
- 今日运动摘要
- 最近重要事件
- 双方状态卡片

### 页面层次

- 首页：总览和快捷入口
- 日历：时间轴和日常记录
- 计划：任务管理和计划拆解
- 运动：健康数据展示
- 我的：设置、权限、同步状态、隐私配置

## 6. 技术方案

## 6.1 推荐技术栈

建议优先采用原生 iOS 技术栈：

- Swift
- SwiftUI
- Observation 或 ObservableObject
- SwiftData 或 Core Data
- HealthKit
- EventKit
- WidgetKit
- CloudKit

### 为什么建议原生

- 接入 HealthKit 和 EventKit 更直接
- SwiftUI 适合快速搭建高质量 iOS 界面
- 与 Apple 生态整合更自然
- 后续做 Widget、Watch、Live Activities 更顺畅

## 6.2 架构建议

建议采用分层架构，而不是将逻辑直接堆在 View 中。

推荐分层：

- Presentation：SwiftUI 页面、组件、状态绑定
- Domain：业务模型、用例、规则
- Data：本地数据库、系统 API 适配、云同步
- Integration：HealthKit、EventKit、CloudKit 等适配层

### 模块拆分建议

- AppCore：基础配置、路由、依赖注入
- CalendarModule：日历与记录
- PlanningModule：待办与计划
- FitnessModule：运动与健康展示
- CoupleModule：情侣关系与共享逻辑
- ProfileModule：用户设置与隐私
- SharedUI：通用组件与设计系统

这种拆分方式有利于未来继续增加新模块，而不会让代码结构失控。

## 6.3 本地存储方案

建议首期采用：

- SwiftData：开发效率较高，适合 iOS 新项目

如果预期数据模型复杂、迁移频繁或未来有更多高级查询需求，可考虑：

- Core Data

### 本地存储原则

- 本地优先，保证离线可用
- 系统 API 数据与应用业务数据分层缓存
- 为同步冲突预留版本字段和更新时间字段

## 6.4 云同步方案

推荐优先考虑：

- CloudKit

优点：

- 对 iOS 原生应用友好
- 和用户 Apple ID 生态一致
- 适合情侣共享这种账号规模相对可控的场景

建议同步策略：

- 用户个人数据默认私有数据库
- 共享任务、共享活动、共享记录进入共享数据库
- 用变更时间和服务端记录版本做冲突处理

如果未来考虑跨平台，再评估：

- Firebase
- Supabase
- 自建后端

## 7. 核心数据模型建议

以下是建议的数据实体，重点是为扩展预留空间。

### 7.1 用户与关系

#### User

- id
- nickname
- avatar
- birthday
- timezone
- createdAt
- updatedAt

#### CoupleSpace

- id
- name
- anniversaryDate
- createdAt
- updatedAt

#### Membership

- id
- userId
- coupleSpaceId
- role
- joinedAt

### 7.2 记录体系

#### Record

- id
- ownerUserId
- coupleSpaceId
- type
- subType
- title
- note
- startAt
- endAt
- value
- unit
- status
- visibility
- metadata
- source
- createdAt

其中：

- `type` 用于区分喝水、排便、月经、睡眠等
- `subType` 用于未来细分
- `metadata` 用于存储扩展字段，避免频繁改表
- `source` 用于标记 manual / healthkit / imported

### 7.3 任务体系

#### Task

- id
- ownerUserId
- coupleSpaceId
- parentTaskId
- planLevel
- title
- description
- status
- priority
- startAt
- dueAt
- isAllDay
- repeatRule
- visibility
- assigneeUserId
- linkedCalendarEventId
- linkedGoalId
- createdAt
- updatedAt

#### Goal

- id
- ownerUserId
- coupleSpaceId
- title
- description
- goalPeriod
- status
- startAt
- endAt
- createdAt
- updatedAt

Goal 用于承接年计划、月计划等较高层级目标，Task 用于落地执行。这能避免所有计划层级都挤在同一张任务表里失控。

### 7.4 运动与健康缓存

#### HealthMetricSnapshot

- id
- ownerUserId
- metricType
- bucketType
- startAt
- endAt
- value
- unit
- source
- createdAt

说明：

- `metricType` 如 steps、heartRate、sleepDuration
- `bucketType` 如 day、week、month
- 用于缓存展示数据，不替代 HealthKit 原始事实来源

## 8. 关键系统集成

## 8.1 EventKit

用途：

- 读取系统日历事件
- 创建和更新应用关联事件

建议做一层 `CalendarSyncService` 统一封装，不要在页面中直接调用 EventKit。

### 处理原则

- 权限申请与功能触发解耦
- 明确同步方向：单向同步或双向同步
- 首期建议以“应用到系统日历”为主，减少冲突复杂度

## 8.2 HealthKit

用途：

- 读取步数、睡眠、心率、活动能量等数据

建议做一层 `HealthDataService`：

- 统一权限申请
- 统一查询接口
- 统一数据映射到应用内部模型

### 处理原则

- 用内部模型隔离外部 API
- 手动记录和 HealthKit 数据分开处理
- 保留来源标识

## 8.3 WidgetKit

可在后续版本加入：

- 今日任务组件
- 今日步数组件
- 情侣纪念日倒计时组件

## 8.4 通知系统

通知建议拆分为：

- 计划提醒
- 喝水提醒
- 睡眠提醒
- 周期相关提醒
- 共享任务提醒

本地通知足够覆盖首期需求，后续如有邀请、共享事件等再接入推送。

## 9. UI / UX 建议

### 9.1 整体风格

建议走“温和、清爽、长期可用”的设计风格，避免过度少女风或过强工具感。因为这是高频应用，长期使用下可读性和轻负担比装饰性更重要。

### 9.2 设计系统建议

- 建立统一色板
- 定义记录类型对应图标和颜色
- 卡片式布局展示摘要信息
- 图表组件统一风格
- 日历标记样式统一

### 9.3 情侣场景体验

- 允许切换查看“我 / TA / 我们”
- 共享内容尽量减少重复录入
- 首页体现关系感，但不要影响核心效率

## 10. 隐私与合规

这是该产品必须重视的一部分。

### 必须明确的点

- 哪些数据只存本地
- 哪些数据会同步到 CloudKit
- 哪些数据会共享给伴侣
- 哪些数据来自 HealthKit

### 必要能力

- 逐项授权说明
- 可撤销共享
- 可导出个人数据
- 可删除数据
- 敏感数据单独可见性控制

## 11. 开发阶段建议

## Phase 1: MVP

目标：先做出最小可用版本。

包含：

- 用户本地使用
- 日历记录
- 基础待办清单
- EventKit 单向同步
- HealthKit 基础读取
- 首页摘要

暂不做：

- 完整情侣共享
- 复杂统计分析
- Apple Watch
- Widget

## Phase 2: 情侣共享

包含：

- CoupleSpace
- 共享任务
- 共享记录
- 共享权限控制
- CloudKit 同步

## Phase 3: 数据洞察

包含：

- 周报 / 月报
- 趋势图表
- 健康与生活记录关联展示
- 个性化提醒

## Phase 4: 生态增强

包含：

- Widget
- Apple Watch
- Live Activities
- 更完整的健康分析能力

## 12. 开发难点与风险

### 12.1 权限复杂度

HealthKit 和 EventKit 都涉及系统权限，需要设计良好的授权引导，否则首屏体验会很差。

### 12.2 同步冲突

情侣共享、本地编辑、云同步和系统日历同步叠加后，冲突会比较复杂。必须在模型层保留更新时间、来源和版本字段。

### 12.3 敏感数据处理

月经、排便、睡眠等数据比普通任务更敏感，产品层面必须提供足够明确的共享控制。

### 12.4 数据模型失控

如果前期直接按功能堆字段，后续拓展到情绪、饮食、体征时会越来越混乱。必须从一开始就做统一的 Record 模型和扩展元数据结构。

## 13. 推荐落地结论

如果目标是做一款长期可迭代、深度接入 iOS 生态的情侣生活记录应用，推荐采用以下路线：

- 前端：SwiftUI
- 数据层：SwiftData 起步，必要时迁移 Core Data
- 系统集成：HealthKit + EventKit
- 云同步：CloudKit
- 架构：模块化分层架构

产品上先围绕这三个基础能力打磨：

- 日历记录
- 多层级任务计划
- 健康运动展示

同时在底层保留三类关键扩展能力：

- 用户与情侣共享空间分离
- 统一 Record / Task / Goal 数据模型
- 外部系统 API 与业务模型解耦

这样后续无论拓展到纪念日、情绪、饮食、AI 总结，还是 Apple Watch，都不需要重做整体架构。
